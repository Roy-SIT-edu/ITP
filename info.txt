5. Why Large Inputs Took So Long
--------------------------------

Current robust sample size after import:
  - sessions: 116
  - time slots: 255
  - rooms: 109

Naive candidate check upper bound:
  116 requirements x 255 slots x 109 rooms = about 3.2 million combinations

Actual filtered candidate assignment count measured:
  98,940 candidate assignments

The original expensive part was not just candidate generation. The old model
builder compared assignments pairwise inside room, staff, and student-group
buckets. On the robust sample, this produced very large pairwise loops:

  room_pair_comparisons_upper: 89,772,330
  staff_pair_comparisons_upper: 111,366,420
  group_pair_comparisons_upper: 123,620,450

Total rough pairwise loop pressure:
  about 324 million comparisons

Example high-candidate rows:
  Workshop / Hybrid / Seminar Room rows often had:
    45 matching slots x 51 matching rooms = 2,295 candidates each

This is why the first large generation attempt ran for much longer than
expected and returned UNKNOWN.


6. Solver Performance And Conflict-Review History
-------------------------------------------------

Main solver files:
  ITP Programming App 2\timetable-app\backend\app\solver\model_builder.py
  ITP Programming App 2\timetable-app\backend\app\solver\cp_sat_solver.py
  ITP Programming App 2\timetable-app\backend\app\services\schedule_service.py

Original route timeouts:
  - Normal generation default: timeout=20 seconds
  - Some auto-resolve paths use timeout=5 seconds and fast_mode=True
  - CP-SAT timeout starts after model building, so slow model building can make
    the user wait far longer than the configured solver timeout.

First speed fix:
  - Replaced expensive pairwise clash variables with bucketed constraints.
  - This made the 116-row robust sample generate successfully in about 25.46s.
  - Result then:
      solver_status: OPTIMAL
      scheduled rows: 116
      hard violations: 0

Regression noticed by user:
  - The user showed that hard-conflict inputs used to still generate a
    timetable for review, but after the first speed fix the page showed:
      solver_status: INFEASIBLE
      0 sessions shown
      0 issues
  - This happened because hard clashes had become strictly blocking, instead
    of reviewable hard violations.

Second solver fix:
  - Restored reviewable hard-conflict behavior using heavy hard-conflict
    penalties.
  - Added regression test:
      test_unavoidable_resource_clash_still_returns_reviewable_schedule
  - Pushed commit:
      0b283f4 Optimize timetable conflict generation
  - Pushed to:
      origin/main

Regression after second solver fix:
  - The robust input started returning UNKNOWN again.
  - Reason: the relaxed hard-conflict penalty model was being used for every
    input, even inputs that could be scheduled cleanly.
  - The penalty model made the search harder and could time out before finding
    or confirming a timetable.

Current solver fix in the working tree:
  - The model builder now accepts:
      relax_hard_conflicts: bool = False
  - Strict mode:
      room/staff/group conflicts are hard linear constraints.
  - Relaxed mode:
      room/staff/group conflicts become heavy penalties.
  - CpSatTimetableSolver now:
      1. Builds and solves the strict model first.
      2. Returns strict result if OPTIMAL or FEASIBLE.
      3. Falls back to relaxed hard-conflict model only when strict result is
         INFEASIBLE.
      4. Leaves UNKNOWN as a timeout/search failure instead of automatically
         falling back.
  - UNKNOWN message changed to:
      "Solver timed out before finding a timetable. Try again or reduce flexible room/time choices."

Current measured result on the 116-row robust input after this fix:
  - Direct solver:
      elapsed: about 24.15s
      status: OPTIMAL
      assignments: 116
      soft_score: 3900 in direct solve context
  - ScheduleService generation:
      elapsed: about 23.47s
      schedule_run_id: 4
      solver_status: OPTIMAL
      hard_violation_count: 0
      soft_score: 6470
  - Latest app database later showed:
      run 5
      status: COMPLETED
      solver_status: FEASIBLE
      scheduled sessions: 116
      hard violations: 0
      soft_score: 6780

Important note on different solver results:
  The user asked why each run can give a different result.
  Explanation:
    - Many valid timetables may have similar or equal score.
    - The solver uses multiple workers:
        num_search_workers = 8
    - Multi-worker CP-SAT search is not fully deterministic.
    - If timeouts are involved, it may return the best timetable found so far.
  To make generation more repeatable:
    - use num_search_workers = 1
    - set a fixed random_seed
    - disable randomized search if enabled
    - add more tie-break penalties such as earlier days/times or stable room
      ordering
  Tradeoff:
    deterministic generation can be slower or may produce slightly worse soft
    scores.
