Timetable Scheduler - Student-Friendly Code Rundown
===================================================

This file explains the current timetable scheduler code after the latest
navigation, review-page, and student-group updates.

Current app shape:
  The main workflow is Import Data -> Generate Timetable -> Review Timetable
  -> Export Timetable. Settings is a separate sidebar tab for configuration.
  Soft constraint priority ranking now lives in Settings, while Generate
  Timetable focuses on running the solver and showing soft preference context.
  The Review page now uses a week-calendar timetable view with Previous Week,
  Next Week, date, time-range, refresh, and display controls.

Latest UI update:
  - Added a Settings sidebar tab.
  - Moved soft constraint Priority Ranking into Settings.
  - Renamed the workflow step from "Priorities & Generate" to
    "Generate Timetable".
  - Made Workflow and Reference Data sidebar groups collapsible.
  - Replaced the Review timetable density grid with a weekly calendar-style
    view that can shift between weeks.
  - Made Student Groups editable in Reference Data and added default
    programme/year groups such as DSC Y1 P1 and DSC Y1 P2. The generated
    group code is internal; admins work with Programme, Year, Group No.,
    and Size.
  - Added a Settings night mode toggle that persists on the device.

File: ITP Programming App 2/timetable-app/backend/app/services/soft_constraint_priority_service.py
Purpose:
  Defines default soft constraints, calculates rank-based weights, and saves
  the configured ranking.

Key snippet:
  def weight_for_rank(rank: int, total: int) -> int:
      return max(1, total - rank + 1) * 5

Student explanation:
  Higher-ranked soft constraints get larger weights. Lower-ranked constraints
  still matter, but the solver is allowed to break them first if tradeoffs are
  required.


File: ITP Programming App 2/timetable-app/backend/app/services/student_group_service.py
Purpose:
  Maintains default student-group partitions by programme and year.

Key snippets:
  DEFAULT_STUDENT_GROUPS_PER_YEAR = 2
  DEFAULT_STUDENT_GROUP_SIZE = 40
  return f"{programme_code.strip().upper()} Y{year} P{partition}"

Student explanation:
  Each programme year now gets default groups like DSC Y1 P1 and DSC Y1 P2.
  Each generated group starts at 40 pax. The code is generated internally, so
  admins do not need to type or remember codes. Existing groups are preserved,
  and admins can add more groups by programme/year when needed.


File: ITP Programming App 2/timetable-app/backend/app/services/database_service.py
Purpose:
  Powers the Reference Data CRUD pages and Excel sync.

Key snippets:
  ColumnSpec(..., options=ROOM_TYPE_OPTIONS, min_value=1, max_length=80)
  ADMIN_DATABASE_TYPE_IDS = ("rooms", "staff", "programmes", "modules", "student-groups")
  if config.id == "programmes":
      ensure_programme_year_groups(db, item)

Student explanation:
  Student Groups is now returned by the database metadata API, so the admin
  page has proper columns for adding or editing groups. Group Code is no
  longer an admin-facing column. When a group is added, the backend generates
  the internal code from Programme, Year, and Group No.; if Group No. is left
  blank, the backend picks the next available partition for that programme
  year. When a programme is added or its year count changes, the app creates
  the missing default P1/P2 student groups for each year.

  Database columns also carry validation metadata. Fixed-value fields such as
  Room Type, Day, and Week Pattern can be rendered as dropdowns.
  Numeric fields such as Capacity, Years, Group No., and Size have min/max
  bounds. Text fields such as room codes, staff names, programme names, and
  module titles have length caps. The backend enforces the same checks for
  manual edits and Excel uploads, so invalid data cannot bypass the UI.
  Campus Mode is not admin-facing; rooms derive it from the Virtual checkbox,
  and requirements derive it from Delivery Mode.
